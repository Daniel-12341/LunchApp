   IMPORTANT: Use `.maybeSingle()` NOT `.single()` — `.single()` throws on 0 results (see RESEARCH.md Pitfall 5). Server actions MUST be in a separate file from client components (RESEARCH.md Pitfall 2).

This component manages all state for Step 1 (meal selection). Step 2 (confirmation) and Step 3 (success) will be added in Plan 02 — for now, the submit button should call a placeholder `handleSubmit` that sets a `pendingOrder` state (Plan 02 will use this to show the confirmation screen).

**State variables:**
- `activeTab: number` (0-3, default 0)
- `selectedItem: { categoryIndex: number, itemIndex: number, useAltPrice?: boolean } | null`
- `specialRequests: string`
- `previousOrder: { meal_name: string, meal_category: string, price: number, customisation: string | null } | null`
- `isSpinning: boolean` (guards Surprise Me)
- `spinDisplayItem: { item: MenuItem, catIdx: number, itemIdx: number } | null` (ticker display during animation)
- `step: 'select' | 'confirm' | 'success'` (default 'select', Plan 02 handles confirm/success)

**Page layout (top to bottom):**

1. **Header**: Fun greeting — e.g. "Hey {name}! What's for lunch?" with emerald colour scheme, food emoji. Use Tailwind for playful styling.

2. **Shortcuts row** (ABOVE tab bar per user decision):
   - "My Previous Order" card: Fetch previous order via `getPreviousOrder(name)` in a useEffect on mount. If data exists, show a card with meal name, category emoji, and price. Tap to re-select: sets `selectedItem` to matching category/item indices (find by meal_name in MENU data) and switches `activeTab`. If no previous order, hide this card entirely.
   - "Surprise Me!" button: Disabled while `isSpinning`. On click, runs ticker animation (see below). Styled as a fun prominent button with dice/sparkle emoji.

3. **Surprise Me ticker display**: Only visible when `isSpinning` or when `spinDisplayItem` is set. Shows the currently cycling item name in a fixed-height box. After animation ends, fades out after ~1 second.

4. **Tab bar**: Four tabs showing `{emoji} {name}` for each MENU category. Active tab highlighted (Claude's discretion on styling — underline, pill, or bold). Tap switches `activeTab`.

5. **Menu items list**: Renders `MENU[activeTab].items`. Each item shows:
   - Name and price (on one line, name left-aligned, price "R{price}" right-aligned)
   - Description below in smaller muted text
   - For Carbonara (has `priceAlt`): Show base price "R140" with a small toggle/label "(+ R2 with cream)" — tapping the toggle switches the price variant. Default: no cream (base price).
   - Selected item: highlighted border/background with checkmark. Tap selected item to deselect. Tap different item to select it (deselects previous).
   - Selection state tracks `categoryIndex` and `itemIndex` so switching tabs preserves selection knowledge, but per user decision "switching tabs deselects previous" — actually re-reading CONTEXT.md: "only one item selected at a time across all categories (switching tabs deselects previous)". Implementation: when user taps an item, it selects. When user switches to a different tab and selects a NEW item there, the old one deselects (this is automatic since selectedItem only stores one). The statement means the visual highlight disappears when you switch tabs (because the selected item is on a different tab). No special logic needed — `selectedItem` naturally handles this.

6. **Special requests field**: `<textarea>` below the menu list. Placeholder text like "Any special requests? (e.g., no onions, extra cheese)". Controlled by `specialRequests` state.

7. **Sticky submit button**: Fixed at bottom of viewport. Shows selected meal name and "R{price}" when an item is selected. Disabled (greyed out) when nothing selected. On tap: sets `step` to `'confirm'` (Plan 02 will render the confirmation screen when step is 'confirm').

**Surprise Me ticker animation:**
