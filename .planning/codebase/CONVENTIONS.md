# Coding Conventions

**Analysis Date:** 2026-03-15

## Naming Patterns

**Files:**
- TypeScript/React: `.tsx` for components, `.ts` for utilities
- Examples: `layout.tsx`, `page.tsx`, `globals.css`
- Pattern: lowercase with hyphens for multi-word filenames (e.g., `globals.css`)

**Functions:**
- Components: PascalCase (e.g., `RootLayout`, `Home`)
- Variables/constants: camelCase (e.g., `geistSans`, `geistMono`)
- Examples from codebase:
  - `export default function RootLayout(...)`
  - `export default function Home()`
  - `const geistSans = Geist({...})`

**Variables:**
- Constants: camelCase (e.g., `metadata`, `eslintConfig`)
- Props: camelCase (e.g., `children`)
- Destructured variables follow PascalCase for types, camelCase for values

**Types:**
- React types use `Readonly` wrapper for immutable props (see `layout.tsx`)
- Type imports: `import type { Metadata }` (explicit type imports)
- Examples: `type { Metadata }`, `React.ReactNode`

## Code Style

**Formatting:**
- No dedicated Prettier config file found — uses ESLint/Next.js defaults
- Single quotes not enforced (template literals used as needed)
- Indentation: 2 spaces (inferred from codebase)

**Linting:**
- ESLint 9.x with Next.js configuration
- Config file: `eslint.config.mjs`
- Rules applied:
  - `eslint-config-next/core-web-vitals` (Web Vitals best practices)
  - `eslint-config-next/typescript` (TypeScript support)
- Run command: `npm run lint` (maps to `eslint`)

**Key ESLint Configuration:**
- Located in `eslint.config.mjs`
- Extends Next.js core-web-vitals and TypeScript configs
- Ignores: `.next/`, `out/`, `build/`, `next-env.d.ts`

## Import Organization

**Order:**
1. External library imports (e.g., `import Image from "next/image"`)
2. React/Next.js imports (e.g., `import type { Metadata }`)
3. Local relative imports (e.g., `import "./globals.css"`)
4. Type imports separated with explicit `type` keyword

**Path Aliases:**
- Configured in `tsconfig.json`
- `@/*` → `./src/*`
- Used for consistent imports (e.g., `@/components`, `@/utils`)
- Not yet used in current minimal codebase

**Example from codebase:**
```typescript
import type { Metadata } from "next";
import { Geist, Geist_Mono } from "next/font/google";
import "./globals.css";

export const metadata: Metadata = {
  title: "Create Next App",
  description: "Generated by create next app",
};
```

## Comments

**When to Comment:**
- Inline comments avoided in favor of self-documenting code
- Component descriptions via JSDoc for exported functions (currently minimal)

**JSDoc/TSDoc:**
- Not currently used in application code
- Recommended for public APIs and complex functions once codebase grows
- Example pattern to follow:
  ```typescript
  /**
   * Component description
   * @param prop Description
   * @returns Description
   */
  export function ComponentName(props: Props) { ... }
  ```

## Function Design

**Size:** Keep functions small and focused

**Parameters:**
- Use object destructuring for multiple parameters
- Example from `layout.tsx`:
  ```typescript
  export default function RootLayout({
    children,
  }: Readonly<{
    children: React.ReactNode;
  }>) { ... }
  ```

**Return Values:**
- Explicitly type return values using TypeScript
- React components return `JSX.Element` implicitly
- Type explicit returns for utilities

## Module Design

**Exports:**
- Use `export default` for main component/function per file
- Named exports for utility functions
- Pattern: `export const metadata: Metadata = { ... }`

**Barrel Files:**
- Not yet used in current structure
- Can implement in `@/` alias directories as codebase grows
- Pattern: `src/components/index.ts` exporting multiple components

## TypeScript Configuration

**Compiler Options:**
- Target: `ES2017`
- Strict mode: `true` (enforces strict type checking)
- Module resolution: `bundler` (Next.js compatible)
- JSX: `react-jsx` (modern React 17+ JSX transform)
- Path aliases configured: `@/*` → `./src/*`
- `isolatedModules: true` — ensures each file can be transpiled independently

**Key Enforcements:**
- `strict: true` — enables all strict type-checking options
- `noEmit: true` — TypeScript doesn't emit JS (Next.js handles compilation)
- `skipLibCheck: true` — skips checking .d.ts files for performance
